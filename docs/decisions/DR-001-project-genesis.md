**日期：** 2026-04-14
**状态：** 已接受
**决策者：** Patrick

## 背景

两个 ICT 交易分析系统已完成开发和初步运行：

### ICT_Advisor（原版）

- **架构：** Bash + Claude CLI "提示软件"
- **运行时间：** 8+ 天生产运行
- **核心资产：** ICT Playbook（22K tokens）、SOP（424 行）、10 个反馈记忆、7 个决策记录
- **核心问题：** 无独立验证、高成本（$1.33/天数据采集）、数据不一致

### ict-advisor-refactory（重构版）

- **架构：** Python 5 层流水线
- **开发时间：** 6 天快速开发
- **核心资产：** KnowledgeBrain 图数据库（300+ 节点）、Guardian 验证层、反馈闭环
- **核心问题：** 生产积累不足、配置半成品、CLAUDE.md 过时、无 A/B 测试

---

## 决策

创建新项目 **Michael**，基于 Harness Engineering 哲学，从两个前身项目取长补短。

### 为什么不在现有项目上继续迭代？

1. **原版架构限制**：纯 prompt 架构无法实现代码级验证、门控、幻觉检测。这不是 prompt 能解决的问题。
2. **重构版基础不牢**：配置系统半成品（`from_env()` 未实现）、CLAUDE.md 与实际不符、步骤模块代码重复——在这个基础上加 Harness 层会越来越复杂。
3. **两者都缺少 Harness 核心**：查询循环、上下文预算管理、会话生命周期管理、分层恢复——这些需要从架构层面设计，不是在现有代码上补丁能实现的。

### 为什么叫 Michael？

致敬 ICT (Inner Circle Trader) 创始人 Michael J. Huddleston。

---

## 设计哲学

遵循 Harness Engineering 十条原则：

1. **模型是不稳定组件** → 所有分析结果都必须经过代码级验证
2. **Prompt 是控制面** → 分层指令系统，有优先级，可缓存
3. **查询循环是心跳** → 每步分析有明确的状态管理和恢复路径
4. **工具是受管理接口** → 三态权限（allow/deny/ask），高风险工具有高密度约束
5. **上下文是工作记忆** → token 预算管理，按需加载知识
6. **错误路径是主路径** → 分层恢复，熔断器
7. **恢复优先续行** → 从断点继续，不重来
8. **多 Agent 分割不确定性** → 数据采集、分析、验证、推送各司其职
9. **验证独立于实现** → Guardian 层不调用模型
10. **制度 > 个人技巧** → 标准化 SOP + 自动化流水线

## 继承策略

### 从原版继承

| 资产 | 形式 | 说明 |
|------|------|------|
| ICT Playbook | 迁移为知识图谱的文本补充 | 保留完整的业务深度 |
| SOP | 集成到指令系统 | 保持集中可查 |
| 10 个反馈记忆 | 导入反馈系统 | 真实修正经验 |
| 7 个决策记录 | 参考但不直接复制 | 保持决策追溯 |
| A/B 测试理念 | 重新设计实现方式 | 保持对比验证能力 |
| 红旗禁止交易条件 | 编码到 Guardian | 明确不交易判断 |

### 从重构版继承

| 资产 | 形式 | 说明 |
|------|------|------|
| 直接 MCP 数据采集 | 架构级继承 | 成本/性能基础 |
| 5 层流水线理念 | 重新设计但保持分层 | 架构骨架 |
| Guardian 验证层 | 继承 + 增强 | 独立验证核心 |
| KnowledgeBrain | 继承 + 优化知识构建 | 知识管理核心 |
| 反馈闭环 | 继承 | 持续改进 |
| pytest 测试方法 | 继承 + 扩展 | 质量保障 |

### 不继承（抛弃）

| 项 | 来源 | 理由 |
|----|------|------|
| Claude CLI 当 API 客户端采数据 | 原版 | 成本/性能无法接受 |
| 硬编码路径 | 两者 | 不可移植 |
| session_type 混乱 | 原版 | 工程质量不达标 |
| 空壳步骤类 | 重构版 | 增加复杂度无收益 |
| 过时的 CLAUDE.md | 重构版 | 误导性 |
| 未实现的 from_env() | 重构版 | 半成品 |
