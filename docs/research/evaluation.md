## 维度 1：Harness 成熟度

> 参照 Harness Engineering 五子系统模型评估

### 1.1 指令系统

| 评估项 | 原版 | 重构版 |
|--------|------|--------|
| 入口指令 | CLAUDE.md 94 行，结构清晰 | CLAUDE.md 158 行，但目录结构已过时 |
| SOP 文档 | 完整 424 行，含工具调用表 | 无独立 SOP，逻辑分散在步骤模块 |
| 冷启动测试 | 依赖 7 个知识文件全量加载（~77KB），启动慢 | 按类别过滤加载，启动快 |
| 指令膨胀 | COMMON_CONTEXT 与 CLAUDE.md/rules.md 部分重复 | 无重复，但步骤模块间 validate_output 重复 |
| **评分** | **★★★☆☆** 有良好意识 | **★★☆☆☆** 有结构但 CLAUDE.md 过时 |

### 1.2 状态系统

| 评估项 | 原版 | 重构版 |
|--------|------|--------|
| 持久化 | SQLite 5 表 + A/B 版本标记 | SQLite + DataManifest + 门控状态 |
| 状态一致性 | session_type 值混乱，有重复记录 | 结构化 dataclass，类型安全 |
| 跨会话传递 | 10 个反馈记忆文件 | FeedbackStore SQLite 表 |
| **评分** | **★★☆☆☆** 有持久化但不一致 | **★★★☆☆** 类型安全但配置半成品 |

### 1.3 验证系统

| 评估项 | 原版 | 重构版 |
|--------|------|--------|
| 验证独立性 | ❌ 由 Claude 自己执行验证 | ✅ Guardian 层代码级独立验证 |

| 验证阻断 | ❌ 无阻断能力 | ✅ FAIL 阻断发布 |
| **评分** | **★★☆☆☆** 验证不独立 | **★★★★☆** 代码级独立验证 |

### 1.4 范围控制

| 评估项 | 原版 | 重构版 |
|--------|------|--------|
| 执行范围 | 每个 cron 任务独立运行，无约束 | 5 步流水线 + 门控 |
| 错误传播 | 无门控，可能发布坏结果 | FAIL/NO_TRADE 终止后续步骤 |
| **评分** | **★☆☆☆☆** 无范围控制 | **★★★☆☆** 有门控 |

### 1.5 生命周期

| 评估项 | 原版 | 重构版 |
|--------|------|--------|
| 会话管理 | 无，每次 claude -p 独立调用 | 流水线生命周期管理 |
| 启动协议 | CLAUDE.md 定义但依赖模型遵守 | run.py 代码强制执行 |
| 交接机制 | 无 | 无 |
| **评分** | **★☆☆☆☆** 基本无管理 | **★★★☆☆** 有生命周期但无交接 |

### 1.6 Harness 维度综合

| 子系统 | 原版 | 重构版 |
|--------|:---:|:---:|
| 指令系统 | ★★★ | ★★ |
| 状态系统 | ★★ | ★★★ |
| 验证系统 | ★★ | ★★★★ |
| 范围控制 | ★ | ★★★ |
| 生命周期 | ★ | ★★★ |
| **综合** | **★★** | **★★★** |

**关键发现：**
- 原版的指令系统（SOP、Playbook）质量高，但 Harness 其他子系统几乎空白

- 两者都没有实现 Harness 核心的"查询循环"和"上下文预算管理"

## 维度 2：性能

| 评估项 | 原版 | 重构版 | 倍数 |
|--------|------|--------|------|
| 数据采集单次成本 | ~$0.19 | $0 | ∞ |
| 数据采集时间 | ~9 分钟 | 秒级 | ~100x |
| 日均采集成本（7 次） | ~$1.33 | $0 | ∞ |
| 日均采集时间 | ~63 分钟 | <1 分钟 | ~60x |
| 分析 Prompt 大小 | ~100KB+（全量加载） | ~29KB（压缩 71%） | 3.4x |
| 单次分析 max_turns | 无限制 | 5 | 可控 |
| 单次分析超时 | 无限制 | 600s | 可控 |
| E2E 单报告时间 | 未测量 | ~393s（优化后） | — |

**评分：** 原版 ★★☆☆☆ → 重构版 ★★★★☆

**关键发现：**
- 重构版在成本和速度上完胜，"直接 MCP 调用"是最大的架构改进
- 重构版 prompt 压缩 71% 效果显著
- 两者都缺少上下文预算管理（prompt 大小随知识增长无约束）

## 维度 3：效率（开发与运维）

| 评估项 | 原版 | 重构版 |
|--------|------|--------|
| **代码量** | ~550 行 Bash + ~100KB Markdown | ~6,500 行 Python + ~1,000 行测试 |
| **认知负担** | 低：改 prompt 即改行为 | 中：需理解 5 层架构和数据流 |
| **新增报告类型** | 在 case 语句加一个分支（~20 行） | 新增步骤模块 + 输出模板 + 展示模板 + Schema（~5 文件） |
| **调试** | 只有日志文件 | logging + dry-run + verbose + 门控日志 |
| **测试** | 无自动化测试 | pytest 覆盖主要路径 |
| **部署** | crontab 手工管理 | 同上 |
| **知识更新** | 编辑 Markdown 文件即可 | 需要重新运行构建工具链（5 步） |
| **问题定位** | 看日志猜测 | 结构化错误 + 异常类 + Guardian 报告 |

**评分：** 原版 ★★★☆☆ → 重构版 ★★★☆☆

**关键发现：**
- 原版的"简洁即力量"：改一行 prompt 就能影响行为，适合快速实验
- 重构版的"结构即力量"：改动重但可测试可追踪
- 两者在部署和运维自动化上都薄弱
- Michael 应在两者之间找平衡：保持知识层的 prompt 灵活性，同时有结构化的流水线

## 维度 4：业务成熟度

| 评估项 | 原版 | 重构版 |
|--------|------|--------|
| **ICT 方法论覆盖** | Playbook 22K tokens：22 种 PDA、4 层 17 入场模型、12 种 Weekly Profile | 知识图谱 300+ 节点：42+ PDA、4 层 20 入场模型、55+ 模型 |
| **SOP 完整度** | 424 行完整 6 步 SOP + TradingView 工具调用表 | 无独立 SOP，逻辑分散在代码中 |
| **规范文件** | report-format-spec (547行) + analysis-sop (424行) | report-format (127行) + pda-analysis (144行) + entry-model (170行) + data-collection (117行) |
| **反馈积累** | 10 个反馈文件（Patrick 真实修正，涵盖分析质量、独立性、Session 行为等） | FeedbackStore（设计完备但缺少真实运行数据） |
| **生产验证** | 8+ 天生产运行，有真实准确率统计 | ~6 天开发，性能优化测试数据 |
| **决策追溯** | 7 个决策记录（DR-001 至 DR-007） | 4 个 ADR + 1 个改革计划 |
| **红旗条件** | 9 条明确的禁止交易条件 | 无独立列表 |
| **A+ 清单** | 8 项，明确定义 | 继承，8 项 |

**评分：** 原版 ★★★★☆ → 重构版 ★★★☆☆

**关键发现：**
- 原版的业务深度来自于 **真实使用和迭代修正**，不只是代码量
- 10 个反馈文件是极有价值的资产——记录了模型在 ICT 分析中的系统性偏差
- 重构版的知识图谱数量上更丰富，但缺少生产验证
- 原版的 SOP 集中可查的形式更适合 AI 和人类阅读

## 维度 5：业务一致性

| 评估项 | 原版 | 重构版 |
|--------|------|--------|
| **方向一致性** | 依赖 Claude 自觉，通过 COMMON_CONTEXT 注入一致性 | ConsistencyChecker 代码检测 Session vs Daily bias 矛盾 |
| **数据真实性** | 无验证，Claude 可能幻觉价格 | HallucinationDetector：PDH/PDL/PWH/PWL 与 OHLCV 比对（1-2 点容差） |
| **评分客观性** | Claude 自评（方向错=0 分门控是好的，但评分本身不客观） | 代码级评分（Scorer 精确/接近/部分阈值：10/25/50 NQ 点） |
| **跨报告连贯性** | COMMON_CONTEXT 注入一致框架 | previous_results 链式传递 + ConsistencyChecker 验证 |
| **数据一致性** | session_type 值不一致（"Asia" vs "asia_pre"，"Long" vs "BULLISH"），有重复记录 | 结构化 dataclass + Enum，类型安全 |
| **模型名称正确性** | 无验证 | 检查入场模型名称是否存在于知识库 |
| **PDA 名称正确性** | 无验证 | 检查 PDA 名称是否存在于知识库 |

**评分：** 原版 ★★☆☆☆ → 重构版 ★★★★☆

**关键发现：**
- 原版依赖"模型自律"，这恰恰违反了 Harness 第一原则（模型是不稳定组件）
- 重构版的 HallucinationDetector 和 ConsistencyChecker 是真正的工程化保障
- 原版的数据不一致问题（session_type 混乱）是"无 Harness"的典型后果
- Michael 必须继承重构版的代码级一致性保障
